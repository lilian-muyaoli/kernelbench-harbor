KernelBench‑Harbor

A harder, fairer, agentic redesign of KernelBench — can a model iteratively write a Triton kernel that beats torch.compile on the architectures people actually serve?

10 modern families · ~132 tasks · pass@2 on GPT-5.5 / Opus-4.8 / Gemini-3.5-flash

§1 · Chosen domain

GPU kernel generation — and why it's a meaningful extension

Domain: write a fast GPU kernel for a given PyTorch module. We extend KernelBench (Stanford, arXiv:2502.10517) into an agentic, modern-ops, beat-the-compiler benchmark.

  • High real value — kernels are the bottleneck of LLM training/serving cost.
  • Frontier-relevant — writing fast GPU code is exactly where models are still weak.

Verification is unambiguous

Ask the model to rewrite the PyTorch reference into a custom Triton kernel; the verifier checks (1) the output matches the reference and (2) how much the runtime improves over torch.compile.

→ correctness + a numeric speedup = a clean, objective pass/fail.

After researching where models still fail, the hard pockets are modern, quantized, fused ops — so that's where we build.

§3 · Redesign & task design

Four main improvements over the original KernelBench

① Agentic

Harbor container; the model compiles, runs, reads errors, resubmits over 4 turns. Harder interaction forces harder tasks.

② Modern SOTA architectures

NSA, MLA, MoE, quant, fused — more complex and state-of-the-art than KernelBench's classic ops.

③ Strong baseline

Gate = beat torch.compile(max-autotune); eager speedup also reported. Correct-but-slow fails.

④ Anti-cheat (pytorch2triton)

AST bans offloading core compute to torch.matmul/softmax/sdpa/cuBLAS; requires a real @triton.jit.

§3 · Redesign & task design

Difficulty = strong baseline + a tricky numerical contract

Fusion alone isn't hard. The gap is the exact dequant / scaling spec the kernel must match, such as:

  • packed asymmetric int4 + zero-points
  • fp8 / int8 block-scaling & requant
  • per-token gather · top-k block selection · MLA absorb

Difficulty tracks baseline strength

We drop weak-baseline ops (vanilla attention, naive SSM) frontier models clear in one shot, and keep families where torch.compile/cuBLAS leave real fusion headroom.

§3 · Redesign & task design

Infra, resource budget & how it scales

Workflow

Harbor (agentic env) · Modal (GPU verify, locked L40S) · OpenRouter (model calls).

Resource budget

4 turns (AMD's arena uses 5) + a ~3 min agent timeout → bounds compute/$ (turns & time).

Scale from one hard task

Hard template → neighbors via parameter grids over real HF configs (model, size, gate/down layer, group size, #experts, dtype). One template → 15–30 tasks.

quant 24 · moe 20 · fp8/mla/w8a8 18 each · ssm 10 · fused/lora 6–9 · nsa 4

Oracles for all 10 families (incl. the hardest) — proof every task is solvable.

§4 · Model results

Finding: capable models turn the feedback loop into fixes

GPT-5.5 — iteration repairs

Reads compiler/profiler feedback and fixes errors over turns — e.g. it lands a passing 1.18× NSA kernel by iterating through compile errors. That's the value the agentic loop adds.

flash — weakest of the three

10% pass@2 — it rarely turns the compiler/profiler feedback into a correct, faster kernel.

The agentic loop pays off when a model can act on feedback — fix the compile error first, then make it fast.

§5 · QC methodology

How I checked task quality: unambiguous, reliable, fair

  • Unambiguous — pass/fail is fully objective: the output matches the reference and beats torch.compile. No judgment calls.
  • Reliable verifier — every family ships a known-good oracle that passes, so the verifier provably accepts correct kernels (no false-negatives) and each task is solvable.
  • Fair — every model gets the same instructions and turn/time budget, and every task has a verified-correct oracle, so all are solvable. Difficulty comes from the operator, not tricks — the tasks are based on real architectures and systems.

Difficulty comes from the operator — every task is oracle-verified solvable.

§6 · Training-signal value

Why this data is valuable to train on

  • Dense, verifiable reward — the pass gate is speedup over torch.compile(max-autotune) (eager also reported); a continuous speedup is a natural RL reward, far richer than pass/fail.
  • Un-gameable — AST anti-cheat + "beat compile" gate; the model must do real work, not call cuBLAS.
  • In the right difficulty band — 10–60%: dense learning signal, neither saturated nor impossible.
  • In-distribution & real — every task is a real deployment shape (real HF configs).
  • Agentic trajectories — multi-turn compile→fix→profile traces are exactly the behavior we want to teach.
§7 · Scaling pipeline

From one hard task to 100K

The loop

① Find a hard task — e.g. a fused vLLM kernel; inspired by the takeaways of current benchmarks (what's still unsolved).
② Probe single-shot → confirm it's hard enough that some attempts fail (not trivially solved).
③ Scale the params → a grid of neighbors.
④ Evaluate agentically on Modal.

Scaling params (real HF configs)

open-source model family · size · layer role (gate / down) · group size · #experts · dtype · sequence length
